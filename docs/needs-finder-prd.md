## 一、背景与问题

Reddit 上有一类经典帖子模式——用户公开喊话"有人能做这个不？""我愿意付钱买这个工具"，由此诞生了 Reddinbox 等专门扫描 Reddit 需求信号的 Micro SaaS。

**国内的问题**：
- 小红书、知乎、微博、抖音每天产生海量用户吐槽与需求表达
- 现有舆情工具（倍市得、数说故事、千瓜数据）全都面向大B品牌方，定价高、功能重
- **没有一款轻量工具**，专门帮助创业者、独立开发者从社交媒体里找到"人们愿意付钱却没人做的需求"
- 创业者只能靠手动刷帖、凭感觉判断，效率极低

## 五、核心功能

### 5.1 需求信号识别（核心算法）

系统需识别以下模式的文本：

**强信号（付费意愿明确）：**
- "我愿意付钱买一个能XX的工具"
- "有没有XX，我真的需要，付费也行"
- "求个能XX的App，跪了"

**中信号（需求明确但未表达付费）：**
- "为什么没有一个工具能XX"
- "有没有推荐XX的软件"
- "市面上有没有解决XX问题的产品"

**弱信号（吐槽/痛点，但未明确需求产品）：**
- "每次XX都好麻烦"
- "XX这个问题真的没人解决吗"
- 高赞吐槽帖（赞数 > 500）

### 5.2 需求聚类

- 将相似需求自动归类，计算该需求在过去 7 / 30 天内的**出现频次**
- 展示"需求热度趋势"（上升 / 稳定 / 下降）
- 按行业 / 场景打标签（效率工具、健康、财务、教育、生活…）

### 5.3 需求详情页

每条聚类需求展示：
- 原始帖子列表（附链接）
- 讨论声量（帖数 + 评论数 + 点赞数）
- 情感分析（用户情绪是焦虑？抱怨？期待？）
- 相关关键词云
- 现有竞品（如果有）

### 5.4 订阅 & 推送

- 用户可订阅关键词 / 行业方向
- 每日/每周 邮件推送新增需求信号
- 支持设置"热度阈值"，仅推送超过一定声量的需求

### 5.5 需求收藏 & 笔记

- 用户可标记感兴趣的需求，添加私人备注
- 支持导出为 Markdown / CSV

## 七、技术方案（草案）

### 7.1 数据采集

| 平台 | 采集方式 | 难点 |
|------|--------|------|
| 小红书 | 非官方 API / 模拟请求 | 反爬虫，需 Cookie 池 |
| 知乎 | 官方 API（部分）+ 模拟请求 | 频率限制 |
| 微博 | 微博开放平台 API + 模拟请求 | 高热内容获取相对容易 |

### 7.2 NLP 处理

- **意图识别**：分类帖子是否包含需求信号（二分类模型）
- **需求抽取**：从帖子中提取核心需求短语（NER / LLM 提炼）
- **聚类**：Embedding 相似度聚类（可用 text-embedding + HNSW）
- **情感分析**：判断帖子情绪倾向

> 建议一期直接调用 LLM API（如混元 / 通义）做 NLP，不自建模型。

### 7.3 技术栈建议

```
后端：Python（FastAPI）
爬虫：Playwright + 代理池
NLP：LLM API（混元 / 通义千问）
数据库：PostgreSQL + pgvector（向量存储）
前端：Next.js
部署：腾讯云 / 阿里云
```

## 八、风险与挑战

| 风险 | 严重程度 | 应对 |
|------|--------|------|
| 平台反爬 / 封号 | 高 | 代理池 + 频率控制 + 多账号 |
| 数据合规（个人信息保护法） | 高 | 只采集公开内容，不存储用户个人信息 |
| LLM 识别准确率不稳定 | 中 | 人工抽样校验 + 迭代 Prompt |
| 竞品入场（现有舆情工具转型） | 中 | 专注轻量 + 低价 + 开发者友好 |
| 平台封杀（小红书 / 知乎）  | 中 | 分散数据源，不依赖单一平台 |
